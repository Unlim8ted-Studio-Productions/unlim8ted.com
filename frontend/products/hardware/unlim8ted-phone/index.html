Context

The repo is the object.

The project is organized around the things needed to make the device real: 3D enclosure files, operating system work, documentation, and earlier PCB development that is now kept as reference.

The current hardware direction uses off-the-shelf modules rather than treating the old custom PCB path as the active assembly route.

01 / Hardware baseline

Known parts first.

CM4 + display + power.

The current path centers on a CM4, Waveshare carrier/display hardware, a LiPo power board, and a battery. The page names the stack without pretending it is finished consumer hardware.

02 / Enclosure

Parts that hold parts.

The case is split into printable and serviceable pieces: shell, covers, internal holders, camera volume, and connector transition geometry.

03 / Runtime

A local shell.

Kiosk interface, not a mockup.

The OS work is represented as a device-specific kiosk surface with local modules and a Python-backed runtime layer.

04 / Archived path

Old PCB work stays visible.

The custom PCB files remain in the repo for context. They are not presented as the active fabrication route for the current build.

OS surface

The interface belongs inside the device story.

The embedded preview is framed as a browser-limited look at the local shell. It stays secondary to the hardware/enclosure story instead of competing with it.
